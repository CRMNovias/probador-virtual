PERSISTENT SESSION AND TOKEN VALIDATION
=======================================

Implementation Date: 2025-11-10
Branch: feature/persistent-session-and-token-validation
Status: Completed ✅

OVERVIEW
--------
Implemented automatic session persistence and token validation to prevent users from having to re-authenticate on every app restart. The system now:
- Saves phone number in localStorage for session restoration
- Validates stored tokens automatically on app startup
- Fetches fresh user data from backend when restoring sessions
- Clears invalid/expired sessions automatically

USER EXPERIENCE IMPROVEMENT
---------------------------
BEFORE:
- User logs in with phone + SMS code
- User closes app/browser
- User reopens app
- ❌ Must enter phone number and SMS code again

AFTER:
- User logs in with phone + SMS code
- User closes app/browser
- User reopens app
- ✅ Automatically logged in if token is still valid
- ❌ Only logs out if token is expired/invalid

TECHNICAL IMPLEMENTATION
------------------------

1. STORAGE KEYS (src/constants/storageKeys.ts)
   - Added PHONE_NUMBER constant to store phone in localStorage
   - Phone is saved alongside auth_token and user_profile

2. AUTH CONTEXT (src/context/AuthContext.tsx)
   - Updated login() method signature to accept optional phone parameter
   - Phone is saved to localStorage during login
   - Phone is cleared during logout
   - Added automatic token validation on app initialization:
     * Reads token from localStorage
     * Calls /user/profile endpoint to validate token
     * Updates user with fresh data from backend
     * Clears session if validation fails (401, network error, etc.)

3. AUTH FLOW HOOK (src/hooks/useAuthFlow.ts)
   - Updated verifyCodeHandler to pass phone to login method
   - Ensures phone number is persisted when user authenticates

TOKEN VALIDATION FLOW
---------------------
App Startup → Check localStorage → Token found?
                                        ↓
                                       YES
                                        ↓
                            Call GET /user/profile
                                        ↓
                            ┌───────────┴───────────┐
                            ↓                       ↓
                         SUCCESS                  FAILED
                            ↓                       ↓
                    Update user data         Clear localStorage
                    Stay logged in           Show login screen

RETRY LOGIC
-----------
Token validation uses the existing apiClient retry mechanism:
- Network errors: 3 retries with exponential backoff (1s, 2s, 4s)
- 401 Unauthorized: No retry, immediate logout
- 5xx Server errors: 3 retries with exponential backoff

SECURITY CONSIDERATIONS
-----------------------
✅ Token validation on every app restart prevents using expired tokens
✅ Backend /user/profile endpoint verifies token authenticity
✅ Failed validation triggers complete logout (all data cleared)
✅ Phone number stored in localStorage is not sensitive (already in user profile)

BACKEND REQUIREMENTS
--------------------
The implementation assumes:
- GET /user/profile endpoint accepts Bearer token in Authorization header
- Endpoint returns 401 for invalid/expired tokens
- Endpoint returns UserProfile data on success

If backend doesn't support this endpoint yet, the feature will gracefully
fall back to local user data (commented code in AuthContext can be uncommented).
